System Overview

A three-step pipeline

  • 1
    Classify — crop a center ROI from the webcam frame and run it through MobileNetV2 to get a class and confidence score
  • 2
    Estimate pose — use a fixed-depth pinhole camera model to back-project the ROI bounding box into 3D space at 0.5 m
  • 3
    Render — project 8 box corners back to pixels with cv2.projectPoints and draw a 3D wireframe box with a floating label tag

Model Architectures

Three approaches, one winner

Custom CNN
ObjectCNN
  • 3-layer conv network
  • 64×64 RGB input
  • BatchNorm + Dropout 0.4
  • Trained from scratch
  • ~1.2M parameters
~100% on 5 classes
Vision Transformer
ObjectViT
  • 8×8 patch embedding
  • 64 patches per image
  • 4 transformer layers
  • Trained from scratch
  • 128-dim embeddings
Slower to converge
Transfer Learning ✦ Deployed
MobileNetV2
  • ImageNet pretrained
  • 224×224 RGB input
  • Fine-tuned classifier head
  • Dropout 0.3
  • Best on noisy web data

Where It Struggles

Confusion tells the story

Glasses — weakest class
Web crawl mixed eyeglasses with drinking glasses. The model had no way to separate them.
Tablet ↔ Phone confusion
Flat rectangles look alike at certain angles. The AR screenshot shows this exact case live.
Root cause
All 5 weak classes used web images only. Webcam-captured classes had near-perfect scores.

Limitations & Next Steps

Honest about the gaps

Current limitations
  • !
    Fixed center ROI — only detects objects at center of frame
  • !
    Fixed depth (0.5 m) — box scale is wrong at very close or far distances
  • !
    No background class — always outputs a class even when nothing is there
  • !
    Prediction flicker — no temporal smoothing between frames
Easy wins next
  • →
    Collect webcam images for the 5 web-only classes to push accuracy above 90%
  • →
    Add a 5-frame history buffer to smooth out prediction flickering
  • →
    Add a background class so the system can say "I don't know"
  • →
    Replace the fixed ROI with a sliding-window or YOLO-based detector
